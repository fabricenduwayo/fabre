reconcile-spring-boot-model-registry-with-h2-experiment — platform explanation fields

=== Difficulty ===
Agents must finish a Java reconcile CLI that joins live HTTP candidates with JDBC evidence while composing validation-run void propagation, same-timestamp calibration replay, waiver replacement/regrant state, operative-run timing, and a second event stream for approval quorum. A newer active waiver can be ineligible because one reviewer fills both roles, forcing fallback to an older fully approved grant; approvals also expire across revoke/regrant epochs. These systems interact before tie-break and applied-waiver reporting, and hidden stores change the lifecycle rather than only the metric values.

=== Solution ===
Implement `/app/reconcile-model-release` in Java: fetch registry candidates; derive operative validation metrics after transitive void propagation; replay calibration, waiver, and waiver-approval events with their timestamp/event-id ordering; bind approvals to the latest valid grant epoch; and require distinct risk and model-owner reviewers. Filter waivers by quorum and pre-run grant timing before selecting the newest eligible grant, then evaluate exact reasons, applied waivers, H2 conflicts, and H2-AUC tie-break, validate the schema, and support alternate JDBC stores.

=== Verification ===
Pytest independently recomputes the decision from the live API and H2 store, including transitive run voiding, same-time replay, reciprocal waiver replacement, latest-grant approval epochs, distinct-reviewer quorum, fallback selection, exact reasons, conflicts, and applied waivers. It reruns the Java CLI against the shipped store and three verifier-owned variant databases, including stale pre-regrant approvals and a dual-failure partial waiver, so static manifests and lifecycle shortcuts fail.
