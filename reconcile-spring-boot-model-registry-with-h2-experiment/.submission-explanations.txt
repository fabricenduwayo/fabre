reconcile-spring-boot-model-registry-with-h2-experiment — platform explanation fields

=== Difficulty ===
Agents must finish a Java reconcile CLI that joins live HTTP candidates with JDBC evidence while composing three stateful governance systems: operative validation runs with supersession voiding, calibration replay that can void a stale snapshot row, and waiver event replay with reciprocal replacement pairs. The shipped data includes registry metric overstatement, voided validation runs, a valid replacement grant, malformed and revoked waivers, and delta's dashboard calibration row that disagrees with replayed calibration_events — so reading only the policy body or snapshot tables promotes the wrong model.

=== Solution ===
Implement `/app/reconcile-model-release` in Java: fetch registry candidates, derive operative validation metrics after voiding superseded runs, replay calibration_events through release_context.decision_at, join version-scoped lineage, and replay waiver_events with reciprocal replacement validation. Suppress only matching raw gate failures, report only grants that were actually used, record H2-canonical conflicts, rank qualifiers by H2 AUC, validate the schema, and write the requested output for either the default or an alternate JDBC store.

=== Verification ===
Pytest recomputes the complete decision from the live API and H2 store, checking schema conformance, operative-run selection, calibration replay, waiver lifecycle, exact reason codes, conflicts, and applied_waivers. It reruns the Java CLI against the shipped store and three verifier-built variant databases so hardcoded manifests, snapshot-table shortcuts, and shallow waiver allow-lists fail.
