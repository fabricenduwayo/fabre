=== Difficulty ===
Agents must finish a Java reconcile CLI that joins live HTTP candidates with JDBC evidence while composing validation-run void propagation, same-timestamp calibration replay, waiver replacement/regrant state, operative-run timing, approval quorum, and optional operative-run anchoring on metric waivers. A waiver anchored to a voided run or granted after its anchor's captured_at cannot suppress failures, and replacement pairs with mismatched anchors are malformed. These systems interact before tie-break and applied-waiver reporting, and hidden stores change the lifecycle rather than only the metric values.

=== Solution ===
Implement `/app/reconcile-model-release` in Java: fetch registry candidates; derive operative validation metrics and run ids after transitive void propagation; replay calibration, waiver, and waiver-approval events with their timestamp/event-id ordering; bind approvals to the latest valid grant epoch; require distinct risk and model-owner reviewers; and enforce operative-run anchoring before selecting the newest eligible grant. Filter waivers by quorum, anchor match, and pre-run grant timing, then evaluate exact reasons, applied waivers, H2 conflicts, and H2-AUC tie-break, validate the schema, and support alternate JDBC stores.

=== Verification ===
Pytest independently recomputes the decision from the live API and H2 store, including transitive run voiding, same-time replay, reciprocal waiver replacement, latest-grant approval epochs, distinct-reviewer quorum, operative-run anchoring, fallback selection, exact reasons, conflicts, and applied waivers. It reruns the Java CLI against the shipped store and four verifier-owned variant databases, including stale pre-regrant approvals, dual-failure partial waivers, and voided-anchor traps, so static manifests and lifecycle shortcuts fail.
