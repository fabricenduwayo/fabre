The model-registry API and the H2 experiment store in this box disagree about several churn-model candidates, and I need the release step finished so we know which model is actually safe to promote. The registry serves candidate metadata and deployment aliases at http://localhost:8080 (it starts on boot; `bash /app/start-registry.sh` brings it back if it ever dies) and is correct as shipped — don't modify it. The canonical validation metrics, feature-hash lineage history, and calibration status live in the H2 database at `jdbc:h2:file:/app/experiment-db/experiments` (user `sa`, empty password); there are no SQL dumps in the image, so JDBC is the only way at that evidence.

Finish the Maven skeleton at /app/reconcile-model-release so that `java -jar /app/reconcile-model-release/target/reconcile-model-release-0.1.0.jar` reads the promotion rules in `/app/policy/promotion-policy.md` together with `/app/policy/promotion-policy-amendments.md` (amendments override the body per section 1.4), pulls the candidates from the API and the evidence from H2, applies every gate, the tie-break, and the missing-evidence rule, and writes `/app/build/release-decision.json` conforming to `/app/schemas/release-decision.schema.json`: the promoted model id (or null), every non-promoted candidate under `rejected` with the policy's exact reason codes, and every API-vs-DB disagreement on an H2-canonical field under `conflicts` with both values. Watch the lineage join — the history table is keyed by model and version, and candidates missing from the experiment store entirely must fail closed the way the policy says.

The jar has to derive the decision from the live sources at run time, not replay a precomputed answer. It also takes an optional H2 JDBC URL as its first argument and an optional output path as its second (defaulting to the store and manifest paths above), and pointing it at a different experiment database must produce the correct decision for that data. Build offline with `mvn -o -f /app/reconcile-model-release/pom.xml package`; the manifest must come from running the jar, not be written by hand.
