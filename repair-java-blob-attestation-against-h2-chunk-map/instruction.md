The object store at `jdbc:h2:file:/app/store/objects` vouches for every object as verified, and that guarantee has failed: customers are retrieving content that no longer matches what they declared, so the store is trusting objects it should be refusing. Finish the control at `/app/attest-objects` that re-establishes each object's integrity independently of the store's own claims, following the attestation policy at `/app/policy/attestation-policy.md`, and build it with `/app/attest-objects/build.sh`.

The store holds each object's bytes in two representations, an ordered chunk map and a materialised `blob_path`, and only one is authoritative while the other drifts. Integrity has to be recomputed against the representation the store treats as canonical, not the one that is easiest to read, so determine which that is rather than assuming the copy that looks like the content is the one it stands behind. The store's attestation cache has drifted the same way and cannot be believed, so each object's status is recomputed from its bytes and reconciled against what the cache asserts, with every disagreement recorded.

The policy defines the verdicts, the digest the store accepts, and the shape of the recorded outcome at `/app/build/attestation-report.json`. A blob-store API on `http://localhost:8080`, with its routes in `/app/blob-store/README.md`, is available for observing how the store serves and re-materialises objects. Optional positional arguments are the JDBC URL then the output path, defaulting to the store above and the report path in the policy, and pointing the control at another store must give that store's answer.
