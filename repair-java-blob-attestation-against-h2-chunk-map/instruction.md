The object store at `jdbc:h2:file:/app/store/objects` attests every object it holds as verified, and that attestation cannot be trusted: customers have retrieved content that does not match what they uploaded, so tampered and damaged objects are being vouched for as sound. The integrity auditor at `/app/attest-objects` is the control meant to catch this, and it is unfinished. Complete it so it independently re-verifies every object and writes an integrity verdict for each to `/app/build/attestation-report.json`, then build it with `/app/attest-objects/build.sh`.

Integrity is judged against the bytes the store treats as an object's canonical content. The store retains two representations of those bytes, an ordered chunk map and a materialised `blob_path`, and only one is authoritative; the other can lag behind, so attesting against the wrong one clears tampered objects and condemns sound ones. Determine which representation the store actually stands behind rather than trusting the copy that merely looks like the content. An object is `intact` only when its canonical content matches its declared digest, `corrupt` when it reads back but does not match, and `unattestable` when the content cannot be read or was declared under a digest the store cannot vouch for, which is anything other than sha256. The attestation cache is itself one of the claims under audit, so treat its status as an assertion to check, never as evidence.

Write `intact`, `corrupt`, `unattestable`, and `conflicts` at the top level. `intact` is a list of object ids; each `corrupt` and `unattestable` entry carries `object_id` and a `reason`, where `corrupt` is `digest_mismatch` and `unattestable` is `missing_content` when the bytes cannot be read or `unsupported_digest` when the declared algorithm is not sha256. Under `conflicts`, record every object whose attested status disagrees with the verdict you reach, each row carrying `object_id`, `cache_status` for what the store asserts (`verified` or `failed`), and `actual_status` for what you found (`intact`, `corrupt`, or `unattestable`). A blob-store API runs on `http://localhost:8080` for observing how the store serves and re-materialises objects, with its routes documented in `/app/blob-store/README.md`. Optional positional arguments are the JDBC URL then the output path, defaulting to the store above and `/app/build/attestation-report.json`, and pointing the auditor at another store must give that store's answer. The report holds verdicts only and must never carry reconstructed object bytes.
