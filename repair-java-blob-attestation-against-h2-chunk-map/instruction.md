The blob store at `/app/blob-store` serves objects over a local HTTP API backed by H2 at `jdbc:h2:file:/app/store/objects`, and its attestation endpoint reports every object as verified, which we know is wrong because customers have pulled back content that does not match what they uploaded. Finish the auditor at `/app/attest-objects`, build it with `/app/attest-objects/build.sh`, and have it write `/app/build/attestation-report.json` sorting every object into `intact`, `corrupt`, or `unattestable` and recording under `conflicts` every object whose cached status disagrees with what you recompute. An object is `intact` only when its durable content matches its declared digest, `corrupt` when it reads back but does not match, and `unattestable` when the content cannot be read or was declared under a digest other than sha256, which the store cannot vouch for.

The store keeps two copies of an object's bytes, an ordered chunk map and a materialised `blob_path`, and they do not always agree, so work out which one it actually treats as durable rather than assuming the copy that looks like the content is the one it stands behind. The API stays up while you work, so ingest an object, change it, and read it back to see which copy survives. Settle the rule that way and reimplement it against the database, because the store's own service is not what runs your auditor.

Write each bucket as object ids, with a short `reason` on the `corrupt` and `unattestable` rows, and each `conflicts` row as the object id, the status the cache asserts, and the status you computed. Optional positional arguments are the JDBC URL then the output path, defaulting to the store above and `/app/build/attestation-report.json`; pointing the auditor at another compatible store has to give that store's answer, not this one's. The report is a set of verdicts, so it must never carry reconstructed object bytes.
