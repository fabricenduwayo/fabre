The object store at `jdbc:h2:file:/app/store/objects` vouches for every object as verified, and that guarantee has failed: customers are retrieving content that no longer matches what they declared, so the store is vouching for objects it should be refusing. Restore the guarantee by re-attesting every object against the attestation policy at `/app/policy/attestation-policy.md`, independently of the store's own claims. The control that does this lives at `/app/attest-objects`; finish it and build it with `/app/attest-objects/build.sh`.

Each object was accepted under a declaration of what it should contain, and the bytes the store kept for it may since have been tampered with or lost, in more than one place. Integrity has to be proven against that declaration rather than taken on trust from whichever copy the store hands back, because a copy that no longer matches the declaration cannot stand for the object while one that still does can. The store's own attestation cache has drifted the same way and cannot be believed, so each verdict is proven afresh from the bytes and reconciled against what the cache asserts, with every disagreement recorded.

The policy defines the verdicts, the digest the store trusts, and the shape of the recorded outcome at `/app/build/attestation-report.json`. A blob-store API on `http://localhost:8080`, with its routes in `/app/blob-store/README.md`, is available for observing how the store serves and re-materialises objects. Optional positional arguments are the JDBC URL then the output path, defaulting to the store above and the report path in the policy, and pointing the control at another store must give that store's answer.
