The blob store at `/app/blob-store` serves objects over a local HTTP API on `http://localhost:8080`, backed by H2 at `jdbc:h2:file:/app/store/objects`, and its attestation endpoint reports every object as verified, which we know is wrong because customers have pulled back content that does not match what they uploaded. Finish the auditor at `/app/attest-objects`, build it with `/app/attest-objects/build.sh`, and have it record an integrity verdict for every object in `/app/build/attestation-report.json`. An object is `intact` only when its durable content matches its declared digest, `corrupt` when it reads back but does not match, and `unattestable` when the content cannot be read or was declared under a digest other than sha256, which the store cannot vouch for. The report carries `intact`, `corrupt`, `unattestable`, and `conflicts` at the top level. `intact` is a list of object ids; each `corrupt` and `unattestable` entry is an object with `object_id` and a `reason`, where `corrupt` is always `digest_mismatch` and `unattestable` is `missing_content` when the bytes cannot be read or `unsupported_digest` when the declared algorithm is not sha256.

The store keeps two copies of an object's bytes, an ordered chunk map and a materialised `blob_path`, and they do not always agree, so work out which one it actually treats as durable rather than assuming the copy that looks like the content is the one it stands behind. The API stays up while you work: `PUT /objects/{id}` ingests bytes, `POST /objects/{id}` replaces an object's content, `GET /objects/{id}` and `GET /objects/{id}/blob` read it back two ways, and `GET /objects/{id}/attestation` returns the cached verdict. Ingest an object, change it, and read it back to see which copy survives, then settle the rule and reimplement it against the database, because the store's own service is not what runs your auditor.

Record under `conflicts` every object whose cached status disagrees with what you recompute, each row carrying `object_id`, `cache_status` for what the cache asserts (`verified` or `failed`), and `actual_status` for what you computed (`intact`, `corrupt`, or `unattestable`). Optional positional arguments are the JDBC URL then the output path, defaulting to the store above and `/app/build/attestation-report.json`; pointing the auditor at another compatible store has to give that store's answer, not this one's. The report is a set of verdicts, so it must never carry reconstructed object bytes.
