The object store at `jdbc:h2:file:/app/store/objects` vouches for every object as verified, and that guarantee has failed: customers are retrieving content that no longer matches what they declared, so the store is vouching for objects it should be refusing. Restore the guarantee by re-attesting every object, independently of the store's own claims. The control that does this lives at `/app/attest-objects`; finish it and build it with `/app/attest-objects/build.sh`.

A reference attestor at `/app/ref/attest-ref` reaches the verdict the store should have reported for every object, and your control has to agree with it on the shipped store and on any other store it is pointed at. It answers for any store built on the schema at `/app/store-db/schema.sql`, and nothing outside the schema's rows and the files they point at enters its decision. The shipped store does not exercise everything the reference decides, so what it would say about stores you are never shown is still your control's contract. The reference will not be on the box when your control runs, so agreement cannot come from calling it; how it decides has to be settled in your control while the reference is still there. The attestation policy at `/app/policy/attestation-policy.md` fixes the verdicts and the shape of the record; the rule that assigns them is the reference's to reproduce.

Run the reference as `/app/ref/attest-ref <jdbc-url> <output-path>`, defaulting to the store above and the report path the policy names. Your control takes the same optional positional arguments, the JDBC URL then the output path, and pointing it at another store must give that store's answer. The store's attestation cache has drifted along with the objects and cannot be believed, so each verdict is proven afresh from the bytes and reconciled against what the cache asserts, with every disagreement recorded.
