The object store at `jdbc:h2:file:/app/store/objects` vouches for every object as verified, and that guarantee has failed: customers are retrieving content that no longer matches what they declared, so the store is trusting objects it should be refusing. Restore that guarantee by re-attesting each object against the attestation policy at `/app/policy/attestation-policy.md`, independently of the store's own claims. The control that does this lives at `/app/attest-objects`; finish it and build it with `/app/attest-objects/build.sh`.

The store holds each object's bytes in two representations, an ordered chunk map and a materialised `blob_path`, and either can drift from what the object was declared to hold. Integrity has to be recomputed against the declaration rather than trusted from whichever copy is easiest to read, because a copy that has fallen behind no longer stands for the object while one that still holds the declared content does. The store's attestation cache has drifted the same way and cannot be believed, so each object's status is recomputed from its bytes and reconciled against what the cache asserts, with every disagreement recorded.

The policy defines the verdicts, the digest the store accepts, and the shape of the recorded outcome at `/app/build/attestation-report.json`. A blob-store API on `http://localhost:8080`, with its routes in `/app/blob-store/README.md`, is available for observing how the store serves and re-materialises objects. Optional positional arguments are the JDBC URL then the output path, defaulting to the store above and the report path in the policy, and pointing the control at another store must give that store's answer.
