The rule that decides an object's verdict is not written down, because the store does not document which of its two copies is authoritative. It exists only as the store's behaviour, and the one corpus the agent can read does not exercise the disagreement. In the shipped store every object with a chunk map has exactly one chunk that is byte for byte what blob_path points at, and every object is declared under sha256, so hashing blob_path and trusting the declared algorithm reproduces the correct report on everything the agent can see. An agent that fixes the report until the sample matches gets a clean green light with no signal left, believes it is finished, and ships an auditor that diverges on most objects of every other store. I measured this: an auditor that hashes blob_path and trusts the declared digest matches the shipped report exactly and then fails all six held-out stores, on the buckets and on the conflict list. The failure is not a missed clause, it is stopping when the evidence says stop.

Three decisions carry the task. The durable content of an object is its chunk map concatenated in ordinal order, and blob_path is only a materialised copy that a later replace leaves behind, so the copy that looks like the content is not the one the store stands behind. The only way to settle that is to exercise the running store, ingest an object, change it, and read it back, rather than assuming from the schema. Then bucket discipline: content that cannot be read is unattestable rather than corrupt, and content declared under a digest other than sha256 is unattestable rather than intact, both of which models collapse into the two-bucket intuition. And the conflict list is the disagreement between the attestation cache and what the content actually hashes to, so it has to be recomputed from the chunk map, not read back from the cache the store already got wrong.

Pytest runs the agent's own compiled auditor through the entry point the instruction names and recomputes the expected report from whichever store the auditor is pointed at, reading the H2 rows and the object files directly, so nothing in the verifier encodes a verdict. Most tests synthesise a store the agent has never seen, each isolating one axis: a stale blob copy that only the chunk map corrects, a blob holding one chunk of a multi-chunk object, content matching a declared sha1, a chunk row pointing at a missing file, a verified cache row over content that recomputes wrong, and one store where those interact. Two further tests check the report is a set of verdicts with no reconstructed bytes and that pointing the auditor at another store yields that store's answer. The store service plays no part at grade time, so parity comes from the agent's code. I cross-checked the Python recompute against an independent Java oracle on all seven stores and they agree.
