The rule that decides an object's verdict is deliberately not written down. The policy fixes what the verdicts mean and what the record looks like, and the rule that assigns them lives in the behaviour of the reference attestor at /app/ref/attest-ref, a compiled reference the agent can run against any store while it works but that is gone when its own control runs. I tried the alternative first and measured it: with every clause spelled out in the policy, both difficulty models passed 10 of 10 runs, because a fully documented rule is transcription. With the rule behavioural, the same models went 0 of 5 and 1 of 5, one run passing cleanly and one more missing a single edge, so the task is hard precisely where it should be and still solvable. The shipped store gives no help on its own: it is single generation throughout, every readable copy matches its declaration, and everything is declared under sha256, so a control fitted to the sample matches the reference on everything visible and falls over on the held-out stores; in the difficulty run all ten trials matched the shipped store and the failures were exclusively on stores the agents never saw.

Four decisions carry the task, and each one has a natural wrong default. A stored copy is only eligible when it reads back at the declared size_bytes, so a stale or truncated copy is set aside rather than hashed, which sends it to missing_content instead of a false digest_mismatch. The chunk map and the materialised blob are parallel alternatives, either of which can vouch for an object, so privileging one, which nearly every failing run did, condemns sound objects. The chunk map itself is only the latest generation of rows, because a re-materialisation appends a new generation and leaves the old rows in place. And only sha256 can attest, with no readable declared-length copy outranking the algorithm question. The conflict list then falls out by reconciling recomputed verdicts against the drifted attestation_cache in both directions.

The verifier recompiles the agent's control from source, recomputes the expected report from whichever store the control is pointed at by reading the H2 rows and files directly, and removes the reference before grading, so nothing in the verifier encodes a verdict and shelling out cannot pass. Six synthesised stores each isolate one axis: a stale blob a sound chunk map corrects, superseded generations, sha1 and md5 declarations, a missing chunk file the blob rescues, cache rows wrong in both directions, and one store where they interact. I cross-checked three independent implementations, the Go reference, the Java oracle, and the Python recompute, on all seven stores and they agree everywhere.
