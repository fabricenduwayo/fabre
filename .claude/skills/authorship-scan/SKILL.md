# Authorship scan

Enforce the human-voice rules in `.cursor/rules/authorship.mdc` on a task's prose before submit, and gate any rewrite. Scan, paste raw output, report PASS/FAIL - do not vouch for text you have not scanned.

## Step 0 - resolve the target task folder (do this first, every run)

The target is one task folder at the repo root /Users/fabrice-mac-mini/Documents/snorkel-ai (kebab-case, contains task.toml). Milestone tasks contain steps/milestone_N/ subfolders and no root instruction.md.

1. If $ARGUMENTS is non-empty, treat it as the target folder name (or path). Use it.
2. If $ARGUMENTS is empty, list the candidate task folders - every directory at the repo root that contains a task.toml, EXCLUDING .review-scratch/, docs/, jobs/, scripts/, tools/, and .git/ - and ask the user which one with AskUserQuestion. If exactly one candidate exists you may confirm it instead of asking.
3. Confirm the chosen path exists and contains task.toml. If it does not, stop and tell the user exactly what you looked for. Never guess or run against the wrong folder.
4. Detect the task type: milestone if steps/milestone_N/ folders exist and there is no root instruction.md; otherwise non-milestone. Remember it - later steps branch on it.

Do not proceed past Step 0 until the target folder is resolved and confirmed.

## Step 1 - identify the prose targets

Build the list of files to scan under the resolved task folder `<task>`:

- Non-milestone: `<task>/instruction.md`.
- Milestone: every `<task>/steps/milestone_N/instruction.md` (one per milestone).
- Plus `<task>/README.md` if it exists.
- Plus `<task>/.submission-explanations.txt` if it exists (hidden dotfile at the task root).

List the real files you found by absolute path. Every scan below runs against each target - rg searches an explicit file path even though the dotfile is hidden.

## Step 2 - run the three pre-submit scans (paste output verbatim)

For each target file, run all three and paste the raw output. Do not summarize what you think it said.

```bash
rg -n $'—|–' <target>                                                              # unicode em/en dash - empty in his prose
rg -in 'furthermore|it is worth noting|in conclusion|additionally|delve into|leverag|game changer' <target>
rg -n 'DRAFT|HUMAN REVIEW REQUIRED|NEEDS HUMAN REVIEW|Co-authored-by|Generated by' <target>
```

Each should return nothing on Fabrice-authored text. Any hit is a FAIL for that file, except the em-dash carve-out in Step 4.

## Step 3 - rewrite gate (only if a rewrite happened or is proposed)

Skip this step if the prose was untouched or only lightly grammar-cleaned. If an agent rewrote, or proposes to rewrite, any target outright, gate it. Run the rewrite check on each affected file:

```bash
rg -in 'really|deliverable|as follows|the following|ensure that|in order to|note that|utilize|robust|seamless|comprehensive' <target>
rg -no '[`\w]+, [`\w]+ (and|or) ' <target>   # dropped Oxford comma - he never does this
rg -n '^\s*[-*#]' <target>                    # bullets and headings
wc -w <target>
```

Then:

- Read at least three other `instruction.md` files elsewhere in the repo and compare voice. They are the reference, not a general sense of good technical writing.
- Report the rewrite-check output, the word count, and a line-by-line note of which sentences are Fabrice's and which are new. Reused sentences should stay reused - replacing a line that was already fine is a regression even if it scans clean.
- A rewrite passes only if BOTH hold: the three Step 2 scans are clean, AND the voice matches those instruction.md files. If either fails, hand the draft back and say exactly why - do not ship it.

## Step 4 - the one exception

Verbatim third-party template/boilerplate lines that a CI check requires to match exactly may keep their em-dash. Nothing else is exempt. If you invoke this carve-out, name the line and the check that pins it.

## Step 5 - report

- Never claim the text "passes the authorship check" without the Step 2 (and, if a rewrite, Step 3) output pasted above.
- Show a compact verdict per target, most severe first: `PASS` / `FAIL` / `NA` + one line of evidence (which scan, which line).
- Then one overall `PASS` / `FAIL` for authorship across all targets.
- End with a short prioritized fix list for every FAIL: file, line, the offending scan, and the concrete change (swap the em-dash for `-`, cut the corporate transition, restore Fabrice's original sentence).
