The release attestation worker at `/app/attest-worker` issues trust decisions for signed release artifacts, and right now it believes the artifact registry instead of the signing evidence. Registry metadata can lag or disagree with what was actually signed, so a stale or tampered registry entry can get an artifact marked trusted. Make the worker decide trust from canonical signing evidence, following `/app/policy/signing-trust-policy.md`.

`artifact_evidence` in `jdbc:h2:file:/app/attestation-db/attestation` is authoritative for an artifact's digest, its signing key, and whether that key is revoked; a revoked signer is denied outright. The detached signature comes from `GET /artifacts/{id}` on the bundled metadata API on port 8080 (`bash /app/start-api.sh` if needed), but the digest sent to `POST /verify` must be the canonical one from H2, never the registry's, and a successful lookup never substitutes for verification. Drain `pending_attestations` and record one row per artifact in `attestation_reports` with verdict `trusted`, `denied`, or `quarantine` plus the reason_code the policy assigns to each API status.

Build with `/app/attest-worker/build.sh` and run `com.snorkel.attest.Main`. It takes an optional JDBC URL; the default is the store above, and reports are always written from and to whichever store it was given.
