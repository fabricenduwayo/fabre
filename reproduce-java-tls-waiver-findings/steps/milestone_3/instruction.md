Milestone 3: produce the final findings. First load /app/config/policy.yaml and /app/config/crypto.toml and validate them against /app/schema/policy.schema.json and /app/schema/crypto.schema.json. Those files carry the operative parameters: the review date, the rotation window, the allowed TLS versions, which environments require mTLS, the trusted issuer set, and the minimum key sizes.

Then take the joined evidence and apply the adjudication precedence the report spells out (see the normative precedence appendix) to decide each service as allow, deny, or rotate, with the matching reason_code and whether a waiver carried it past the protocol-violation check. Set `waiver_applied` true whenever an in-force, in-scope, same-type waiver excused the violation on the adjudication path — including hygiene rotations (`cert_near_expiry`, `weak_key`, `untrusted_issuer`) where that cover was still in force. A waiver is in force only when its status is granted and `waiver_expires_on` is on or after the `review_date` in policy.yaml — do not treat a separate expired status string as authoritative. The precedence was amended this cycle, so follow the report's amended ordering rather than a naive top-to-bottom reading: where it matters, a service that trips more than one rule must be reported under the rule the amended precedence puts first. The allow/deny/rotate totals should line up with the review's Findings Overview.

Run this stage as `com.mariner.audit.Main validate` with no arguments — it is the final pipeline stage, not a generic schema-validation utility that takes file paths. With no extra arguments it must read the configs and the milestone 2 joined evidence itself, validate the configs against their schemas as part of its work, and write the result to /app/out/findings.json, which must validate against /app/schema/findings.schema.json.
