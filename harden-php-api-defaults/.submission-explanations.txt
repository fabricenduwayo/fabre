DIFFICULTY

Agents must reconcile a long amended hardening standard against live PHP, SQLite, and multi-worker credential state. A generation cutover stages a successor rather than immediately replacing current: health confirmations from both distinct allowed origins must be accumulated atomically before activation, while same-origin repeats, disallowed origins, concurrent confirmations, higher-generation replacement, stale pending fencing when the published generation advances again, existing predecessor state, and audit reasons all interact. This couples CORS resolution, token persistence, cutover ordering, audit output, and cross-worker locking instead of leaving them as independent patches.

SOLUTION

The oracle replaces the stale PHP configuration, HTTP helpers, ledger writer, front controller, and launcher. Its owner-only state envelope stores current, predecessor, and pending successor digests plus generation and confirmation state under one cross-process lock. Initial bootstrap activates directly; later bootstraps stage or replace pending successors, allowed-origin health calls build quorum, and the second distinct origin atomically activates the successor and creates the two-use predecessor window. A published generation advance beyond an unfinished successor fences that stale pending credential until a higher-generation bootstrap stages its replacement. Audit writes still reconcile `audit_log` transactionally, preserve legacy ids with NULL origins, and ignore shadow tables.

VERIFICATION

Pytest drives the real HTTP service through 20 focused tests and 45 seeded randomized lifecycles. It checks distinct-origin confirmation, same-origin non-quorum, disallowed pending denial, unfinished-successor replacement, stale pending denial after generation advance, OPTIONS preflight exclusion from the ledger, exactly one activation under concurrent workers, activation audit reasons, predecessor handoff, and the prior CORS, ledger, live-input, corrupt-state, and disclosure controls. Every randomized response and audit row is compared with a hidden state machine. Oracle receives 1.0 and no-op receives 0.0.
