DIFFICULTY

Agents must reconcile an amended hardening standard across PHP file state and a live SQLite ledger. Sponsorship has phases, secret rotation clears in-flight progress, and an allowed-origin invalid-token denial now revokes only that origin's sponsorship. Every state mutation remains gated by its audit append under the credential lock. The denial fence composes request classification, origin resolution, phase freshness, rollback, and later re-sponsorship without disturbing another origin or an existing confirmation.

SOLUTION

The oracle stores the normalized secret fingerprint in the documented `pending_secret_digest` envelope field and re-reads it on every health request under the token lock. A mismatch clears progress and rebinds pending state; first confirmation removes stale sponsorship for unconfirmed origins. Before an `invalid_token` denial is published, it removes the resolved allowed origin from sponsorship state inside the same audit transaction wrapper. Insert failures never call the publisher, token-write failures roll back the ledger, and activation still rechecks generation before publication.

VERIFICATION

Pytest drives the real HTTP service through 25 behavioral tests and 45 seeded randomized lifecycles. A targeted probe sponsors both origins, submits an invalid credential from one, proves only that sponsorship was fenced, and completes cutover after re-sponsorship. Existing probes cover secret rotation and SQLite append failures during sponsorship, confirmation, activation, and predecessor consumption. Randomized lifecycles now apply the denial fence while comparing every response and audit row with the reference state machine.
