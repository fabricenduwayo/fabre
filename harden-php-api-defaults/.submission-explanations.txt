DIFFICULTY

Agents must reconcile a long amended hardening standard against live PHP, SQLite, and multi-worker credential state. A later-generation successor cannot confirm at an origin until the incumbent current credential sponsors that same origin after staging; sponsorship and confirmation sets are generation-bound, survive concurrent workers, and are cleared by replacement. Activation then partitions predecessor overlap into one atomic allowance per trusted origin. These state machines interact with stale-pending fencing, CORS resolution, audit output, and a malformed-state bootstrap hard gate that precedes both secret and generation evaluation.

SOLUTION

The oracle replaces the stale PHP configuration, HTTP helpers, ledger writer, front controller, and launcher. Its owner-only state envelope stores current, predecessor, and pending successor digests plus generation, sponsorship, confirmation, and remaining predecessor-origin allowances under one cross-process lock. Current-token health requests register post-staging origin sponsorship; only a sponsored pending token can confirm there, and the second sponsored origin atomically activates it. Each allowed origin can then accept the predecessor once; repeated, absent, and disallowed origins fail without consuming the other allowance. Higher-generation replacement clears pending sets, stale pending credentials cannot be sponsored, and malformed existing state always returns `409` before generation or secret checks.

VERIFICATION

Pytest drives the real HTTP service through 21 focused tests and 45 seeded randomized lifecycles. It checks unsponsored denial, post-staging per-origin sponsorship, pre-staging non-credit, replacement clearing, stale-pending fencing, concurrent single activation, per-origin predecessor consumption under concurrency, and malformed-state refusal after a large generation advance, plus the prior CORS, ledger, live-input, and disclosure controls. Randomized requests force sponsor-confirm and origin-bound predecessor sequences and compare every response and audit row with a hidden state machine. Oracle receives 1.0 and no-op receives 0.0.
