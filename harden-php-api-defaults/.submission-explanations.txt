DIFFICULTY

Agents must reconcile a long amended hardening standard across PHP file state and a live SQLite ledger. Sponsorship now has phases: the first confirmation invalidates earlier sponsorship for every unconfirmed origin, and live bootstrap-secret rotation clears both sponsorship and confirmation progress without removing the pending successor. Every health transition that mutates credential state must also append its audit row before publishing the new envelope, under the same credential lock. These rules compose temporal state, secret re-reads, database migration, failure rollback, and multi-worker activation.

SOLUTION

The oracle stores a normalized secret fingerprint with pending state and re-reads it on every health request under the token lock. A mismatch atomically clears progress and rebinds the pending envelope; first confirmation also removes sponsorship for unconfirmed origins. The audit writer exposes a transaction wrapper that reconciles and inserts `audit_log`, invokes token publication while SQLite remains uncommitted, then commits. Insert failures never call the publisher, and token-write failures roll back the ledger. Activation rechecks generation immediately before this gated publication, then creates per-origin predecessor allowances.

VERIFICATION

Pytest drives the real HTTP service through 24 behavioral tests and 45 seeded randomized lifecycles. New probes pre-sponsor both origins before first confirmation, rotate the live secret during a partial cutover, and install an aborting SQLite trigger while sponsorship, confirmation, activation, and predecessor consumption are attempted. They verify failed appends publish no state, then remove the trigger and confirm the same progress or allowance remains usable. Randomized sequences include wrong-phase confirmation and mid-cutover secret rotation while comparing every response and audit row with the reference state machine.
