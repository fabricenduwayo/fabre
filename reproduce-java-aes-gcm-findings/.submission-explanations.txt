Difficulty Explanation
----------------------
This task is hard because the rules you need are buried in a long forensic report, not in a summary table. The agent has to build a Java pipeline that reads those appendices, walks SQLite audit history in recorded_at order with rotation-replacement precedence, pulls ciphertext out of a GIF, and runs AES-256-GCM with frame_id as AAD. The audit timeline is adversarial: one frame is rotated twice (the latest rotation wins, not the first), another is rotated down to a lower version number (so assuming versions only increase picks the wrong key), and two frames carry explicit report nonce overrides (transcribing only the first derives a wrong nonce for the second). Get the key version or nonce wrong on one frame and authentication fails even if the rest of the code is fine.

Solution Explanation
--------------------
I built a three-stage Java CLI under com.mariner.forensic.Main. The rules stage regex-parses Appendix C and D for the precedence lists and every report nonce override (frm-003 and frm-006). correlate applies that logic to the SQLite audit events: latest rotation_replacement for frm-002/frm-004/frm-005 (including frm-004's two rotations to v4 and frm-005's rotation down to v2), and override-or-derived nonce per frame. decrypt extracts ciphertext from the GIF and verifies AES-256-GCM using keys from the key_material table. The oracle compiles to /app/pipeline/classes and runs each stage in order. The main gotchas are taking the most recent key_rotated event (not the first), using replacement_key_version as-is even when it is numerically lower, and reading both nonce overrides from Appendix D.

Verification Explanation
------------------------
Each milestone runs the compiled Main subcommand for that stage and checks JSON against the published schema plus hidden expected files under tests/. M1 checks precedence order, the frm-003 override, and rules ground truth — derived_nonce_rule must match the verbatim Appendix C labeled line (trailing period optional) and both Appendix D overrides must be present. The pipeline must compile Main under /app/classes or /app/pipeline/classes. M2 checks JDBC correlation in recorded_at order, including the multi-rotation and rotation-to-lower-version cases and the per-frame nonce source. M3 checks schema-valid findings, every in-scope frame with auth_ok and reason_code authenticated, and an exact match to hidden expected_findings.json.
