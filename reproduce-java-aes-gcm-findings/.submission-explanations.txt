Difficulty Explanation
This is a three-milestone AES-GCM forensic pipeline. The revised corpus removes the analyst outcome tables, leaving source-authority records, conflicting candidate catalogues, separate status journals, signed revision chains, and withdrawn errata. Agents must resolve immutable candidate IDs through incorporated review revisions and board dockets, then combine those rules with raw SQLite lifecycle histories and GIF evidence instead of copying per-frame conclusions.

Solution Explanation
Milestone 1 parses rule and exception candidate definitions, follows status or disposition events, restricts them through revisions and dockets incorporated by FND-06, reads the configured derivation, and joins only admitted frames to SQLite override bytes. Its oracle installs only the rules implementation. Milestone 2 overlays the solved correlator, rebuilds, orders events by effective_at, recorded_at, and event_id, and preserves connected surviving provenance. Milestone 3 overlays the decryptor, rebuilds, parses the GIF application extensions, authenticates with frame_id as AAD, and emits either an authenticated hash or an auth_failed record with a null hash.

Verification Explanation
Each milestone verifier reruns its Java stage, validates the published JSON Schema, and compares all base records against hidden expected fixtures. Milestone 1 also excludes tokens and frames belonging only to superseded, rejected, withdrawn, or unincorporated candidates. Milestone 2 installs verifier-private assignment events to prove recorded_at resolves an effective_at tie and event_id resolves a full timestamp tie, asserting the winning event provenance before restoring the database. Milestone 3 temporarily changes one correlated nonce to force a real GCM tag failure and checks auth_ok=false, reason_code=auth_failed, and a JSON-null plaintext hash before restoring the artifact.
