# Snorkel Terminus / Harbor Task Rules

When helping me with this project, follow these rules.

## instruction.md style

* Do not write instruction.md in a polished AI/GPT style.
* Keep my wording and tone as much as possible.
* Do not use “You are an expert programmer…” or similar synthetic language.
* Do not add emojis.
* Do not add heavy markdown, long bullet lists, or long sections.
* Keep the instruction short and natural, like a real engineer asking a coding agent for help.
* Length should sit between one sentence and three short paragraphs.
* Give the agent the goal and requirements, but do not give step-by-step solution hints.
* Use absolute paths for any file paths.
* No canary string in instruction.md (it means an old skeleton was used).
* The task should be interesting/useful and noticeably unique from Terminal-Bench 2/3 and Terminus Edition 1.
* If my wording is unclear, ask me instead of inventing details.
* Do not rewrite everything from scratch unless I ask.
* Light grammar cleanup is okay, but my writing should still sound like me.

## Task files

Follow the Terminus/Harbor format:

* instruction.md
* task.toml
* environment/Dockerfile or environment/docker-compose.yaml
* solution/solve.sh
* tests/test.sh
* tests/test_outputs.py

For milestone tasks the layout is different (per-milestone `steps/milestone_N/` with its own solution and tests) — ask me before assuming a milestone layout.

## task.toml rules

* Include all required metadata: difficulty, category/task_type, subcategories (empty if none), number_of_milestones (0 if none), codebase_size, languages, tags (3-6 keywords).
* Set difficulty from the measured accuracy over 5 runs on the two frontier models, not a guess.
* Set allow_internet = false under [environment].
* Declare realistic cpus, memory_mb, storage_mb, and agent/verifier/build timeouts. The oracle must pass within them.

## Dockerfile rules

* Keep environment files inside environment/.
* Digest-pin every FROM (and any compose image:) with @sha256. No floating tags.
* Use a sanctioned/canonical Terminal-Bench base image for the final runtime stage; if non-canonical, add a short written justification in the Dockerfile or README.md.
* Install tmux and asciinema — the agent runtime needs both or all runs fail.
* Pin application dependency versions (pip, npm, apt, etc.) and pin any downloaded binary by version + sha256.
* Include a .dockerignore for any non-trivial task; prefer narrow `COPY src/ src/` over `COPY . .`.
* Keep environment/ at or below 100 MiB total and 50 MiB per file.
* Store source as real files, not heredocs. Do not run apt-get upgrade.
* Use multi-stage builds to keep build tools out of the runtime image, unless the task itself requires the agent to compile.
* Do not copy solution/ or tests/ into the Docker image.
* Do not use privileged mode.
* Include required runtime tools if needed by the task environment.
* Keep dependencies reproducible.

## solution/solve.sh rules

* The oracle solution must be deterministic.
* The solution should be human-written; minimal LLM help for syntax is okay.
* Do not hardcode the final answer.
* Show the command sequence that solves the task.
* Use set -e where appropriate.
* Do not rely on internet access during the oracle run.

## tests rules

* tests/test.sh is required.
* tests/test.sh must run Python pytest tests (even for non-Python tasks — drive the CLI/service/files from pytest).
* tests/test.sh must write /logs/verifier/reward.txt (always, including on failure/error).
* The reward if/else/fi block is the canonical end of test.sh — do not add a trailing exit after it (check_test_sh fails on this).
* Do not install packages or download anything inside tests/test.sh.
* Put test dependencies in the Dockerfile.
* Every test needs an informative docstring (CI checks this).
* Tests must pass ruff lint.
* Tests should check behavior and final correctness, not just source-code text patterns.
* No order-dependent tests and no hardcoded random values (seed or assert on properties).
* Tests must match what instruction.md asks for, and everything tested must be stated or clearly implied in instruction.md.
* Do not test requirements that are not stated or clearly implied in instruction.md.

## General behavior

* Explain important changes before making them.
* Do not make the task hard only by adding too many edge cases.
* Keep the task multi-step and terminal/environment based.
* If something may violate Terminus rules, warn me before editing.

# Snorkel Terminus / Harbor Cursor Rules

These rules apply whenever you help with Snorkel Terminus / Harbor task creation.

This project is about creating Terminal-Bench style tasks where an AI agent works in a terminal environment. The work usually includes `instruction.md`, `task.toml`, `environment/Dockerfile`, `solution/solve.sh`, `tests/test.sh`, and `tests/test_outputs.py`.

Do not treat this as Beaver or PaperBench. This rules file is only for Terminus / Harbor tasks.

## My working style

- Keep my own wording and tone as much as possible.
- Do not make the writing sound polished, generic, or AI-generated.
- If I write a rough instruction, lightly clean grammar only.
- Do not rewrite everything from scratch unless I ask.
- If something is unclear, ask me instead of inventing details.
- Explain important changes before making them.
- Warn me if something may violate Terminus rules.

## Good first task types for me

Prefer tasks in:

- software-engineering
- debugging
- build-and-dependency-management
- data-processing
- system-administration

Avoid starting with:

- UI tasks
- multi-container tasks
- security-heavy tasks
- ML-heavy tasks
- very large codebase tasks
- milestone tasks until I am comfortable

## Required regular task files

For a regular non-milestone task, use this structure:

```text
task-name/
├── instruction.md
├── task.toml
├── environment/
│   └── Dockerfile
├── solution/
│   └── solve.sh
└── tests/
    ├── test.sh
    └── test_outputs.py
```

Optional files are okay when needed, such as data files, config files, README.md, or custom tools.

Do not include `rubric.txt` in the ZIP. Rubrics are generated and edited in the Snorkel Platform submission UI.

## Milestone task warning

Milestone tasks do not use root-level `instruction.md`, `solution/`, or `tests/`.

They use:

```text
steps/milestone_N/instruction.md
steps/milestone_N/solution/
steps/milestone_N/tests/
```

Only use milestone structure when the task is truly a milestone task.

## instruction.md rules

The `instruction.md` file should sound like a real engineer asking a coding agent for help.

Must be:

- concise
- clear
- human-sounding
- well specified
- interesting or useful
- unique
- written with absolute paths

Do not make it GPT-style.

Avoid:

- “You are an expert programmer...”
- emojis
- heavy markdown
- long sections
- long bullet lists
- step-by-step walkthroughs
- hints
- exact solution values
- telling the agent which commands to run
- canary strings
- relative paths
- prompts that are mostly hard because of too many edge cases

The prompt should give the agent the **what**, not the **how**.

Good instruction workflow:

1. I write a rough version first.
2. Lightly clean my wording.
3. Keep it short and natural.
4. Make sure every requirement has a matching test.
5. Make sure tests do not enforce anything not stated or clearly implied.

## task.toml rules

`task.toml` must include the required metadata and limits.

Include:

- `author_name`
- `author_email`
- `difficulty`
- `category`
- `subcategories`
- `number_of_milestones`
- `codebase_size`
- `languages`
- `tags`
- `expert_time_estimate_min`
- `junior_time_estimate_min`

Use `number_of_milestones = 0` when there are no milestones.

Use about 3 to 6 descriptive tags.

Each task should have exactly one main category/task type.

Subcategories can be multiple if the task truly spans multiple challenge areas. Leave blank if none apply.

Runtime limits must include:

```toml
[verifier]
timeout_sec = ...

[agent]
timeout_sec = ...

[environment]
build_timeout_sec = ...
cpus = ...
memory_mb = ...
storage_mb = ...
allow_internet = false
```

Usually set `allow_internet = false` unless the project rules and task design clearly allow otherwise.

For custom docker-compose / multi-container tasks, use both:

```toml
custom_docker_compose = true
is_multi_container = true
```

## Environment and Dockerfile rules

The environment definition must stay inside the `environment/` folder.

Use:

- `environment/Dockerfile`, or
- `environment/docker-compose.yaml` only when needed

Dockerfile must:

- install `tmux`
- install `asciinema`
- pin application dependencies such as pip/npm packages
- use reproducible dependencies
- keep the environment minimal
- use `.dockerignore` for non-trivial environments

Do not:

- copy `solution/` into the Docker image
- copy `tests/` into the Docker image
- use privileged mode
- mount `docker.sock`
- use `SYS_ADMIN`
- use `NET_ADMIN`
- use `SYS_MODULE`
- rely on live internet downloads during oracle or verifier runtime

All solution dependencies should be preinstalled in the environment.

All test/verifier dependencies should be preinstalled in the Dockerfile.

`tests/test.sh` must not install packages or download anything.

## Oracle solution rules

The oracle solution lives at:

```text
solution/solve.sh
```

It proves the task is solvable.

The oracle solution must:

- reliably complete the task
- pass the tests through the Oracle Agent
- be deterministic
- show the command sequence
- derive the answer instead of just writing expected output
- use fail-fast behavior like `set -e` where appropriate
- avoid silent failures
- use fixed seeds if randomness is needed

The oracle solution must not:

- hardcode final answers
- just echo expected output
- use random values without seeds
- depend on time-sensitive behavior
- depend on latency-sensitive behavior
- depend on external internet calls
- be flaky

Useful commands:

```bash
harbor run -a oracle -p <task-folder>
harbor run -a oracle -p <task-folder> -v
harbor tasks start-env -p <task-folder> -i
```

If the Oracle Agent cannot solve the task, the task is broken.

## tests/test.sh rules

`tests/test.sh` is required in every regular task.

It must:

- be the main verifier entrypoint
- run Python pytest
- write `/logs/verifier/reward.txt` or `/logs/verifier/reward.json`
- always write a reward even when tests fail
- avoid package installs
- avoid network downloads

Harbor reads `reward.txt` by default to determine pass/fail.

Binary reward only:

- `1` means success
- `0` means failure

No partial rewards.

Verifier logic must be identical for oracle and agent runs.

## tests/test_outputs.py rules

All verifier tests must be Python pytest tests.

Use Python pytest even for Java, Node, Go, Bash, Rust, or other language tasks.

For non-Python tasks, use Python tools such as:

- `subprocess`
- API clients
- file checks
- process checks
- database clients

Each test needs an informative docstring.

Tests should:

- verify behavior
- verify final correctness
- cover explicit requirements
- cover important implicit behavior
- cover critical edge cases
- match `instruction.md`

Tests should not:

- only check that a file exists
- only check JSON/CSV format
- parse source code looking for brittle patterns
- require hidden behavior not stated or clearly implied
- install packages
- download anything
- use different logic for oracle vs real agent

Ground truth and expected outputs used in tests must live only in `tests/`, not in `environment/`.

The agent must not be able to read the tests during its attempt.

## Rubric rules

Rubrics are handled in the Snorkel Platform submission UI.

Do not include `rubric.txt` in the ZIP.

Rubrics should:

- check meaningful agent behavior
- check trajectory quality
- award useful steps
- penalize harmful, unsafe, incorrect, or wasteful behavior
- include negative criteria when needed

Rubrics should not:

- reference files in `/tests/`
- duplicate unit tests
- say “instruction.md says...” or “task.toml says...”
- describe criteria by file reference instead of behavior

Good negative criterion:

```text
Agent accesses /app/secret/, -1
```

Bad criterion:

```text
Agent does not access /app/secret/, +1
```

## Task design rules

A good Terminus task must be:

- multi-step
- testable
- unique
- standalone
- terminal/environment-based
- runnable without more user input after start
- hard for meaningful engineering reasons

A good task should require:

- multiple commands
- intermediate state handling
- reasoning across files, outputs, commands, or system state

Do not make the main task:

- pure data structures/algorithms
- solvable by one simple command
- mostly a long list of edge cases
- dependent on privileged Docker
- a Long Context task solvable by grep or keyword search
- uninteresting or irrelevant to developers

Acceptance expectations:

- Oracle solution passes
- no-op/NOP should not pass
- real agents should not pass 100 percent across tests
- multiple medium-severity failures means the task should not be accepted

Use this general flow:

# Terminus submission hardening

These are the specific things we missed on `reproduce-java-tls-waiver-findings`
(first pass and reviewer return), `repair-cpp-setup-auditor`, and
`harden-php-api-defaults`. They complement `terminus.mdc` and `final-check.mdc`
(general format) - this file is the "don't make these mistakes again" list. Full
write-up: `docs/terminus-lessons-learned.md`.

## Reading eval feedback (the real blockers)

The `revision_notes` line **"AutoEval execution failed"** is usually **boilerplate**
for submissions that never reached a human reviewer. **Ignore it.** Read the
**right side of the Web UI** (or `stb submissions feedback` → Summary / LLMaJ).
Almost every bounce is one of these three:

### 1. Oracle solution failed

- Local `harbor run -a oracle` green is necessary but **not sufficient** — platform
  Docker/env differences, unpinned deps, or slow apt mirrors can still fail the oracle.
- Download the **difficulty-check artifact** from the summary window; the job log
  names the exact failing command, file, or test.
- Fix, re-run oracle locally, then resubmit.
- **Always bump the ZIP** before re-upload: change something in the task (e.g.
  `# platform-revision: YYYY-MM-DD-rN` comment in `environment/Dockerfile`) or
  the platform may serve a cached build.

### 2. Difficulty is TRIVIAL or EASY

- Diversity gate accepts **MEDIUM or HARD only**.
- If **worst-model pass rate > 60%**, the task is bounced as EASY/TRIVIAL.
- If `languages` includes **`python`**, the bar is **HARD** (best or worst ≤ 20%
  under the OR rule).
- Fix with **real interacting logic**, not relabeling `task.toml` and not a long
  list of trivial edge cases.

### 3. Solvability — "some tests not passed by any agent run"

Two distinct causes:

- **Killer test:** one or more tests passed by **0/10** agent runs. Find the test
  in the per-test breakdown; fix the **spec gap** in `instruction.md` or fix an
  over-strict test. Instruction sufficiency FAIL often shows up here.
- **Verifier timeout:** tests are passable but the verifier ran out of budget on
  **5+ of 10** attempts (≥ 6/10 must finish within budget). Trim bloat: heavy
  image builds before agent work, one milestone soaking the agent budget, installs
  in `test.sh` (forbidden anyway), cold pip/cargo/uv chains.

SHOULD NOT DO:
- Do not treat "AutoEval Failure" as the diagnosis or ask Terminus Bot to explain
  it — the bot is literal and will chase the wrapper, not the LLMaJ summary.

## Submission and review flow

SHOULD DO:
- Default every `stb submissions update` / submit to `--no-send-to-reviewer`.
  Sending to a reviewer is a separate, final step the user explicitly approves.
- Run the oracle to green (`harbor run -a oracle -p <task>`) before *every*
  (re)submission - no edit is too small to re-verify.
- Check assignment state is `NEEDS_REVISION` before any revision work.
- Make all content, explanation, and rubric edits in **one** pass while in
  `NEEDS_REVISION` (any `create_submission` flips to `EVALUATION_PENDING` and
  locks further ZIP uploads).
- On revision, upload the ZIP **and** set platform fields in the **same**
  submission action. Draft text locally first (e.g. `.submission-explanations.txt`),
  then include explanations + rubrics in the single payload that also carries
  `upload_a_zip_file`. The `stb` CLI update only merges the ZIP — inject
  `difficulty_explanation`, `solution_explanation`, `verification_explanation`,
  `test_rubrics`, and `checkbox_evaluate_rubrics` via API in that same call, or
  fill them in the UI before submit.
- Fill all three explanation fields before every submit/resubmit — reviewers
  reject blank ones. Short paragraphs in your own words: why it is hard for an
  agent, the high-level pipeline approach, how per-milestone (or final) tests
  confirm correctness. Applies to milestone tasks too.
- Populate rubrics in the platform `test_rubrics` field (not in the ZIP):
  - **Non-milestone:** flat list, at least 3 negative criteria.
  - **Milestone (`number_of_milestones` > 0):** one `# Rubric N` block per
    milestone; each block needs at least one negative line and a positive-point
    total between 10 and 40. Set `checkbox_evaluate_rubrics = true`.

SHOULD NOT DO:
- Do not send to reviewer to "just see what happens" - there is no self-service
  recall from `REVIEW_PENDING`; only an admin can return it to `NEEDS_REVISION`.
- Do not try to edit a task in `REVIEW_PENDING` / `EVALUATION_PENDING`.
- Do not expect a `rubric.txt` in the ZIP to satisfy the rubric check.
- Do not push explanations/rubrics via API **before** the ZIP upload on a
  revision — a payload-only `create_submission` locks the assignment and the
  platform returns 403 on a follow-up ZIP upload until an admin returns it to
  `NEEDS_REVISION`.

## task.toml

SHOULD DO:
- Set `difficulty` to a real measured value (`easy`/`medium`/`hard`) and meet the
  gate: minimum MEDIUM, and HARD is required when `languages` includes `python`.
- List in `languages` only what the agent writes or the oracle uses. The pytest
  verifier is always Python and does not count.
- Set `codebase_size` from the actual `environment/` file count (~<20 files =
  `minimal`, more = `small`).

SHOULD NOT DO:
- Do not leave skeleton placeholders like `difficulty = "unknown"`.
- Do not add verifier-only `python` to `languages` (it falsely triggers the
  "python => HARD" gate).
- Do not relabel difficulty to pass the gate - genuinely harden the task instead.

## Environment / Dockerfile

SHOULD DO:
- Pin the platform's canonical base image, even for non-Python tasks (install
  php/g++/cmake/jdk on top):

```dockerfile
FROM public.ecr.aws/docker/library/python:3.13-slim-bookworm@sha256:01f42367a0a94ad4bc17111776fd66e3500c1d87c15bbd6055b7371d39c124fb
# platform-revision: YYYY-MM-DD-rN   # bump on every resubmit to bust platform cache
```

SHOULD NOT DO:
- Do not bump the base digest to a "newer" public one - it fails with
  `manifest verification failed`.
- Do not re-upload the same ZIP bytes without a small task change — platform
  caching can mask fixes.

## tests/test.sh

SHOULD DO:
- End the file with the canonical reward block, `rc=$?` immediately before the
  `if`, nothing after `fi`:

```bash
PYTHONPATH=/tests python3 -m pytest --ctrf /logs/verifier/ctrf.json -p no:cacheprovider -rA /tests/test_outputs.py -q
rc=$?
if [ "$rc" -eq 0 ]; then
    echo 1 > /logs/verifier/reward.txt
else
    echo 0 > /logs/verifier/reward.txt
fi
```

- Write `/logs/verifier/reward.txt` on every exit path (early `echo 0; exit 0` is
  fine for build/startup failures *before* pytest).

SHOULD NOT DO:
- Do not add a trailing `exit "$RC"` after the reward block (`check_test_sh`
  fails).
- Do not put logic between `rc=$?` and the `if`.
- Do not install packages or download anything in `test.sh`.

## codebase_applicability gate (build-from-scratch tasks)

The AutoEval `codebase_applicability` check fails the build when `environment/`
contains no source code. It crashes with `is_real_code_base=False` ("only config
files, schemas, JARs, SQL, fixtures, a report - no .py/.java/.js representing a
software project") -> BUILD FAILED -> the check shows `passed=False` on every eval.
Every other gate (difficulty, instruction sufficiency, tests, tb_check,
test_quality_judge, claude_code_reviewer) can be green while this one alone holds
the task in `NEEDS_REVISION`.

SHOULD DO:
- For any task where the agent writes the whole program from scratch, ship a
  minimal **non-solution scaffold** in `environment/` (e.g. `environment/pipeline/`
  with a `Main` dispatcher, stubbed stage classes that throw
  `UnsupportedOperationException`, and a `build.sh`) and `COPY` it into the image.
  It makes the env a "real codebase" without leaking answer logic, and it removes
  the "agent hallucinated a pipeline" robustness failures AutoEval otherwise sees.
- Keep the scaffold compilable (verify with `javac` in the task image) and confirm
  the oracle still passes - oracles that `rm -rf /app/pipeline` then copy their own
  tree are unaffected.
- Point `instruction.md` at the scaffold ("extend the provided scaffold at
  /app/pipeline") instead of "build from scratch".

SHOULD NOT DO:
- Do not put real stage logic, thresholds, or expected values in the scaffold - it
  ships in the image and would leak the solution / lower difficulty.

## Verifier robustness

SHOULD DO:
- Accept any reasonable location the agent could pick for build outputs (e.g.
  search `/app/classes`, `/app/pipeline/classes` for compiled Java), OR state the
  exact required path/command/filename in `instruction.md`.
- Assert on behavior and final correctness, not on incidental file layout.

SHOULD NOT DO:
- Do not hardcode an undocumented path/convention the agent cannot infer - it can
  fail every agent run while the oracle still passes.

## Instruction sufficiency

SHOULD DO:
- Name every control/requirement the tests verify (e.g. `AC-TOKEN-STORE`, CORS,
  bootstrap order, audit logging) without giving the solution; or keep it
  uniformly "reconcile the whole standard at <path>".
- Keep requirements and tests in lockstep: everything tested is stated or clearly
  implied, and everything stated has a test.
- Write instructions in **agent-facing** terms. Say that pipeline stages must be
  reproducible from compiled classes (name the `Main` subcommands) and that JSON
  outputs must not be hand-written. The agent never sees the verifier — do not
  phrase requirements as "the verifier reruns …" or "the verifier checks …".

SHOULD NOT DO:
- Do not half-enumerate controls (naming some implies the list is complete and
  leaves a silent gap for the ones you tested but didn't mention).
- Do not describe verifier behavior in milestone `instruction.md` files.

## Making it harder the right way

SHOULD DO:
- Add real interacting logic/state: e.g. sudo sticky command-tag semantics,
  active authorized-key parsing, or a grant/rescind/replacement waiver lifecycle
  that needs ID pairing and timeline reasoning.

SHOULD NOT DO:
- Do not raise difficulty with a long list of trivial edge cases - it is a
  reviewer red flag and does not raise real difficulty.

### docs/terminus-lessons-learned.md

# Terminus task lessons learned

Practical notes from revising three tasks on the Snorkel Terminus / Harbor
platform:

- `reproduce-java-tls-waiver-findings` (milestone task, Java)
- `repair-cpp-setup-auditor` (regular task, C++)
- `harden-php-api-defaults` (regular task, PHP)

This is the "things that actually bit us" list. Read it before starting a new
task so we do not repeat the same mistakes. It complements the general format
rules in `.cursor/rules/terminus.mdc` and `.cursor/rules/final-check.mdc`; this
file focuses on the concrete failures and the fix for each.

Each item is written as: **Symptom -> Root cause -> Fix -> How to avoid it.**

---

## 0. The three cliffs (read the UI, not "AutoEval Failure")

Expert platform note — treat this as the default triage order after every eval.

### 0.1 "AutoEval execution failed" is usually noise

- **Symptom:** `revision_notes` says `AutoEval execution failed` / build FAILED.
- **Root cause:** Default wrapper when the submission did not reach a human
  reviewer. The **real reason** is in the Web UI summary (difficulty, solvability,
  instruction sufficiency, per-test breakdown) or in `stb submissions feedback`.
- **Fix:** Ignore the wrapper line; triage using section 0.2 below.
- **How to avoid it:** Do not ask Terminus Bot to explain "AutoEval Failure" — it
  chases the boilerplate and misses the LLMaJ summary.

### 0.2 Oracle solution failed (cliff #1)

- **Symptom:** Reference oracle not 100% on platform; difficulty-check build FAILED.
- **Root cause:** Local oracle passed but platform env differs (Docker, deps, apt
  mirrors, image cache).
- **Fix:** Download the difficulty-check artifact from the summary window; read the
  job log for the failing command/test. Fix, `harbor run -a oracle` locally, bump
  ZIP (`# platform-revision: …` in `environment/Dockerfile`), resubmit.
- **How to avoid it:** Oracle green before every upload; never skip re-verify after
  "small" edits.

### 0.3 Difficulty TRIVIAL / EASY (cliff #2)

- **Symptom:** `Difficulty: ❌ EASY` or `TRIVIAL — Requires at least MEDIUM`.
- **Root cause:** Worst-model pass rate **> 60%** on frontier agents. If
  `languages` includes `python`, need **HARD** (best or worst ≤ **20%**).
- **Fix:** Harden with interacting logic/state — not relabeling `task.toml`, not
  edge-case spam. Re-run agent eval after hardening.
- **How to avoid it:** Measure on platform; do not guess difficulty from local runs.

### 0.4 Solvability — tests not passed by any agent (cliff #3)

- **Symptom:** `Status: ❌ Unsolvable` / some tests **0/10**.
- **Root cause (killer test):** Spec gap or broken test — one test never passes.
  Check per-test breakdown; common pattern is instruction sufficiency FAIL (requirement
  tested but not stated in `instruction.md`).
- **Root cause (verifier timeout):** Tests are passable but verifier times out on
  **5+ of 10** attempts (need **≥ 6/10** within budget). Agent cap ~30 min;
  verifier budget is separate.
- **Fix (killer):** Name the requirement in `instruction.md` or relax the test.
- **Fix (timeout):** Trim image bloat, costly milestones, installs in `test.sh`,
  cold dependency chains in the agent path.
- **How to avoid it:** Everything tested is stated or clearly implied; keep env lean.

## 1. Submission and review workflow (the most expensive mistakes)

### 1.1 We accidentally sent everything to a human reviewer

- **Symptom:** All three tasks jumped to `REVIEW_PENDING` when we only wanted
  AutoEval / static checks.
- **Root cause:** `stb submissions update` (and the submit flow) defaults to
  **send-to-reviewer**. We did not pass the opt-out flag.
- **Fix:** Always pass `--no-send-to-reviewer` on every `update`/submit unless
  the user explicitly says "send it to the reviewer now".
- **How to avoid it:** Treat "send to reviewer" as a separate, deliberate, final
  step. Default to checks-only.

### 1.2 There is no self-service recall from `REVIEW_PENDING`

- **Symptom:** Once in `REVIEW_PENDING`, we could not pull the tasks back to fix
  them.
- **Root cause:** The CLI/UI has no "recall" action. Only an admin can move a
  submission back to `NEEDS_REVISION`.
- **Fix:** Ask an admin (via Slack) to return the task(s) to `NEEDS_REVISION`.
- **How to avoid it:** See 1.1. Do not send to reviewer until checks pass and the
  user confirms.

### 1.3 Submissions are only editable in `NEEDS_REVISION`

- **Symptom:** Updates and rubric edits were rejected; we saw "Duplicate
  submission detected" or `403 Forbidden` on ZIP upload.
- **Root cause:** `REVIEW_PENDING` and `EVALUATION_PENDING` are locked states. A
  `create_submission` call flips the task to `EVALUATION_PENDING` immediately, so
  a *follow-up* ZIP upload or rubric call then fails.
- **Fix:** Do all edits (content **and** rubric **and** explanations) while the
  task is in `NEEDS_REVISION`, in one submission action.
- **How to avoid it:** Check state first. Batch everything into one pass.

### 1.3a Do not push platform fields before the ZIP on a revision

- **Symptom:** Explanations and rubrics saved, but instruction fixes stayed
  local; `stb submissions update` failed with "must be NEEDS_REVISION"; direct
  S3 upload returned `403 Forbidden`.
- **Root cause:** We called `create_submission` via API for explanations/rubrics
  only. That locked the assignment in `EVALUATION_PENDING` before the revised ZIP
  uploaded. The platform will not accept a ZIP upload until an admin returns the
  task to `NEEDS_REVISION`.
- **Fix:** Wait for `NEEDS_REVISION`, then upload ZIP and platform fields
  together in one `create_submission` (single payload with `upload_a_zip_file`
  plus explanation/rubric fields). Draft text locally first; do not trigger
  evaluation until the ZIP is in that same payload.
- **How to avoid it:** Revision order: (1) all local edits + oracle green, (2)
  draft `.submission-explanations.txt` and rubric text, (3) one combined update
  that uploads the ZIP **and** sets `difficulty_explanation`,
  `solution_explanation`, `verification_explanation`, `test_rubrics`, and
  `checkbox_evaluate_rubrics`. Never API-push explanations/rubrics alone first.

### 1.4 Rubric is a platform field, not a file, and an empty one fails CI

- **Symptom:** CI failed with "No rubric found" on the non-milestone tasks.
  Reviewer sent back the Java TLS milestone task for a missing rubric even after
  AutoEval passed.
- **Root cause:** The rubric lives in the submission payload (`test_rubrics`),
  not in the ZIP. There is no `rubric.txt`.
- **Fix:** Populate the rubric in the submission form ("Generate your Rubric(s)"
  checkbox), or inject it programmatically into `test_rubrics` via the `stb`
  Python API while the task is in `NEEDS_REVISION` — in the same submission
  action as the ZIP upload (see 1.3a).
- **Rubric shape:**
  - **Non-milestone:** flat list with **at least 3 negative criteria**.
  - **Milestone (`number_of_milestones` > 0):** one `# Rubric N` block per
    milestone; each block needs at least one negative line and a positive-point
    total between 10 and 40. Set `checkbox_evaluate_rubrics = true`.
- **How to avoid it:** Add "rubric populated (correct shape for task type)" to the
  pre-submit checklist. Never assume the zip carries the rubric.

### 1.5 Always run the oracle before every (re)submission

- **Symptom:** A resubmission failed tests that a quick oracle run would have
  caught.
- **Root cause:** Resubmitting on the assumption that "a small edit can't break
  it".
- **Fix:** `harbor run -a oracle -p <task-folder>` must pass green *every* time
  before we update the submission.
- **How to avoid it:** No edit is too small to re-verify. Oracle first, submit
  second.

### 1.6 Explanation fields are required — blank ones get sent back

- **Symptom:** AutoEval passed, but a human reviewer returned the Java TLS task
  because Difficulty, Solution, and Verification were all blank.
- **Root cause:** The three fields live in the submission payload, not the ZIP.
  We submitted content without filling them (or filled them in a separate API
  call that did not survive / was done out of order).
- **Fix:** Draft all three before every submit/resubmit. Keep a local copy in
  `.submission-explanations.txt`. Include them in the same `create_submission`
  payload as the ZIP upload.
- **What to write** (~1 paragraph / 4-6 sentences each, your own words, engineer
  tone — not polished AI prose):
  - **Difficulty** — why the task is hard for an agent (not just humans).
  - **Solution** — high-level pipeline approach and key insight.
  - **Verification** — how the tests (per-milestone or final) confirm correctness.
- **How to avoid it:** Treat blank explanation fields as a blocker on the
  pre-submit checklist. Milestone tasks need them too.

## 2. `task.toml` pitfalls

### 2.1 `difficulty` must be a real measured value

- **Symptom:** `difficulty = "unknown"` was rejected (Java).
- **Root cause:** Leftover skeleton default.
- **Fix:** Use `easy` / `medium` / `hard`, set from the measured accuracy over
  the frontier-model runs.
- **How to avoid it:** Never ship a skeleton placeholder. Grep the toml for
  `unknown` before zipping.

### 2.2 The difficulty gate is enforced, and it is language-dependent

- **Symptom:** C++ came back `TRIVIAL` (needs >= MEDIUM); Java came back `EASY`
  (needs >= MEDIUM); PHP came back `MEDIUM` but failed "requires HARD for python".
- **Root cause:** There is a minimum difficulty gate. Tasks measured below it are
  rejected. **Python-language tasks must be HARD.**
- **Fix:** If the measured difficulty is under the gate, genuinely harden the
  task (see section 6); do not just relabel the toml. For Python, either reach
  HARD or remove Python from `languages` if it is verifier-only (see 2.3).
- **How to avoid it:** Decide the target difficulty up front and design enough
  real, interacting complexity to hit it. (if need ask the use to confirm if you can test this against opus or gpt this project has ways to this this) but this should done maybe after we have done a couple of submissions and we can figure out the level of difficulty

### 2.3 `languages` lists only what the agent/oracle uses, not the verifier

- **Symptom:** PHP and Java tripped the "python => HARD" gate even though the
  agent never wrote Python.
- **Root cause:** We listed `python` in `languages`, but Python was only the
  pytest verifier. The gate keys off `languages`.
- **Fix:** Remove verifier-only languages. PHP became `["php", "bash", "sql"]`;
  Java became `["java", "bash", "sql"]`. The pytest verifier does not count.
- **How to avoid it:** `languages` = languages the **agent** writes or the
  **oracle** solution uses. The verifier is always Python and is implied.

### 2.4 `codebase_size` must match the actual file count

- **Symptom:** `codebase_size = "minimal"` was wrong for a ~22-file environment
  (Java).
- **Root cause:** Mislabeled size; `minimal` is for very small trees (~<20
  files), `small` for larger.
- **Fix:** Count the files actually shipped in `environment/` and set the band
  accordingly (Java -> `small`).
- **How to avoid it:** Set `codebase_size` after the environment is final, not
  from the skeleton.

---

## 3. Environment / Dockerfile

### 3.1 Use the platform's canonical base image digest - do not "upgrade" it

- **Symptom:** `manifest verification failed` in AutoEval after we pinned a
  newer public digest.
- **Root cause:** The platform only trusts its sanctioned/canonical base digest.
  A "newer" public digest for the same tag is not accepted.
- **Fix:** Pin the canonical image used across all our tasks:

```dockerfile
FROM public.ecr.aws/docker/library/python:3.13-slim-bookworm@sha256:01f42367a0a94ad4bc17111776fd66e3500c1d87c15bbd6055b7371d39c124fb
```

- **How to avoid it:** This Python base is sanctioned even for non-Python tasks
  (PHP, C++, Java) because it runs the pytest verifier; install language runtimes
  (php, g++/cmake, jdk) on top of it. Do not bump the digest to chase a newer
  build.

### 3.2 `codebase_applicability` fails build-from-scratch tasks with no source

- **Symptom:** Task stuck in `NEEDS_REVISION` even though difficulty (MEDIUM),
  instruction sufficiency, tests, `tb_check`, `test_quality_judge`, and
  `claude_code_reviewer` all pass. The only red child is `codebase_applicability`,
  failing on *every* eval with `is_real_code_base=False` -> `BUILD FAILED`.
- **Root cause:** The checker scans `environment/`; if it sees only configs,
  schemas, JARs, SQL, fixtures, and a report (no `.py`/`.java`/`.js`), it decides
  the task is "not a real software project" and exits non-zero. Build-from-scratch
  tasks ship zero source by design, so they trip this deterministically.
- **Fix:** Ship a minimal **non-solution scaffold** in `environment/` (e.g.
  `environment/pipeline/`: a `Main` dispatcher, stage-class stubs that throw
  `UnsupportedOperationException`, a `build.sh`, a short README) and `COPY` it into
  the image. Point `instruction.md` at it ("extend the provided scaffold"). No
  answer logic in the scaffold. Verify it compiles in the task image and that the
  oracle (which `rm -rf /app/pipeline` then copies its own tree) still passes.
- **Bonus:** it removes the "agent hallucinated a pipeline / context overflow on
  the from-scratch setup" robustness failures AutoEval otherwise attributes to the
  task.

## 4. `tests/test.sh`

### 4.1 The reward block has a strict canonical shape

- **Symptom:** Static check `check_test_sh` failed.
- **Root cause:** Two issues we hit: (a) a trailing `exit "$RC"` after the reward
  block, and (b) extra logic between `rc=$?` and the `if`.
- **Fix:** End the file with exactly this, with `rc=$?` *immediately* before the
  `if`, and nothing after `fi`:

```bash
PYTHONPATH=/tests python3 -m pytest --ctrf /logs/verifier/ctrf.json -p no:cacheprovider -rA /tests/test_outputs.py -q
rc=$?
if [ "$rc" -eq 0 ]; then
    echo 1 > /logs/verifier/reward.txt
else
    echo 0 > /logs/verifier/reward.txt
fi
```

- **How to avoid it:** Early failure paths (build failed, service didn't start)
  may `echo 0 > reward.txt; exit 0` *before* pytest, but the final pytest reward
  block is the literal end of the file. Always write `reward.txt` on every exit
  path. Never install or download inside `test.sh`.

## 5. Verifier robustness (do not encode hidden requirements)

### 5.1 Do not hardcode an undocumented path or convention the agent can't infer

- **Symptom:** All 10 Java agent runs failed even though the logic was correct.
- **Root cause:** The verifier required `Main.class` at `/app/pipeline/classes`,
  but agents reasonably compiled to `/app/classes`. That exact output path was
  never stated in `instruction.md`, so the test enforced a hidden convention.
- **Fix:** Make the verifier flexible - search the common compiled-output
  directories under `/app` (e.g. `/app/classes`, `/app/pipeline/classes`) instead
  of one hardcoded path. Alternatively, state the exact required path in
  `instruction.md`.
- **How to avoid it:** A test may only enforce what the instruction states or
  clearly implies. If correctness depends on a specific path/command/filename,
  either accept reasonable variants or document it explicitly.

### 5.2 Milestone tests are tightly coupled - keep early checks robust

- **Symptom:** A brittle Milestone 1 check cascaded into later milestone failures.
- **Root cause:** Milestones share cumulative state in the same container, and
  later graders depend on earlier artifacts.
- **Fix:** Keep each milestone verifier tolerant of legitimate placement of build
  outputs and intermediate files; assert on behavior, not on incidental layout.
- **How to avoid it:** Test the milestone's *outcome*, not where the agent put a
  scratch file.

---

## 6. Instruction sufficiency (LLM-as-judge)

### 6.1 Every tested control must be discoverable from the instruction

- **Symptom:** Instruction Sufficiency FAIL on PHP - agents missed the
  `AC-TOKEN-STORE` control (token hashing + owner-only file permissions).
- **Root cause:** We named some controls but not others, which implied the list
  was complete and left a "silent gap" for a control the tests still checked.
- **Fix:** Explicitly name every control area the tests verify (e.g.
  `AC-TOKEN-STORE`, CORS policy, bootstrap ordering, audit logging, debug-
  disclosure), without giving the solution.
- **How to avoid it:** Two valid styles - (a) enumerate all tested control areas,
  or (b) keep it uniformly "reconcile the entire standard at <path>". Never
  half-enumerate. Every requirement tested must be stated or clearly implied; and
  everything stated must have a matching test.

### 6.2 Write instructions for the agent, not the verifier

- **Symptom:** Reviewer asked us to reword all three milestone instructions.
  AutoEval did not catch it.
- **Root cause:** Instructions said things like "the verifier may rerun
  `com.mariner.audit.Main decode` from the compiled classes". The agent never
  sees the verifier — that phrasing leaks grader implementation into the prompt.
- **Fix:** Rephrase in agent-facing terms: each stage must be reproducible from
  compiled classes (name the `Main` subcommands), and JSON outputs must not be
  hand-written. Keep the intent; drop "verifier reruns / checks" wording.
- **How to avoid it:** On every milestone `instruction.md`, grep for
  "verifier" before zipping. If it appears, rewrite for the agent.

---

## 7. Making a task harder the right way

- **Symptom:** Tasks measured too easy; the temptation is to pile on edge cases.
- **Root cause:** Edge-case spam does not raise *real* difficulty and is a
  reviewer red flag.
- **Fix:** Add genuinely interacting policy/logic. Examples that worked:
  - **C++:** sudo *sticky command-tag* semantics (`NOPASSWD:` tags persist across
    entries until overridden) and *active* authorized-key-line parsing (markers
    that disable a key). These force real state tracking.
  - **Java:** a waiver lifecycle where an old waiver is granted, rescinded, then
    *replaced* by a new waiver for the same service - forcing the agent to pair
    rescissions by `waiver_id` and reason over a timeline, not just by service.
- **How to avoid the wrong path:** Difficulty should come from reasoning across
  files/state/time, not from a long list of trivial corner cases.

---

## 8. Housekeeping

- Keep ground truth and expected outputs only in `tests/`, never in
  `environment/`. Do not copy `solution/` or `tests/` into the image.
- Clean junk before zipping: `__pycache__/`, `.pytest_cache/`, `.ruff_cache/`,
  `.DS_Store`, stray `jobs/` or scratch dirs.
- Keep a `.dockerignore` (we had to add `**/node_modules/`) so the build context
  stays small.

## 9. Fast pre-submission checklist (the deltas that bit us)

1. Oracle passes locally (`harbor run -a oracle -p <task>`), green, this run.
2. `task.toml`: `difficulty` is real (not `unknown`); meets the gate; Python is
   HARD if listed.
3. `languages` excludes verifier-only Python; lists only agent/oracle languages.
4. `codebase_size` matches the actual environment file count.
5. Base image is the canonical digest (`sha256:01f42367...`), not a bumped one.
6. `test.sh` ends with the canonical `rc=$?` reward block; reward written on all
   paths; no trailing `exit`; no installs/downloads.
7. Verifier does not depend on an undocumented path/convention.
8. `instruction.md` names every tested control area; every requirement has a
   test, and every test maps to the instruction.
9. No "verifier" wording in milestone instructions — agent-facing reproducibility
   language only.
10. Three explanation fields drafted and non-blank (difficulty / solution /
    verification); local copy in `.submission-explanations.txt`.
11. Rubric populated in the platform with the correct shape:
    - non-milestone: flat list, >= 3 negative criteria;
    - milestone: one block per milestone, each with >= 1 negative line and
      10–40 positive points; `checkbox_evaluate_rubrics = true`.
12. Assignment is `NEEDS_REVISION` before a revision; ZIP upload and platform
    fields go in one `create_submission` (never API-push explanations/rubrics
    before the ZIP — see 1.3a).
13. Submit with `--no-send-to-reviewer`. Only send to reviewer after checks pass
    and the user confirms.
