The worker has to decide trust from two sources that disagree on purpose. Canonical digest and signer revocation live in artifact_evidence in H2, while the registry API serves a registry_digest that can lag what was actually signed, so the obvious path of verifying whatever GET hands back is wrong for art-beta. Revocation has to short-circuit before the API is called at all, and a successful lookup does not stand in for verification, because the detached signature is only checked on POST /verify and only against the H2 digest. Nine GET and POST status rows each map to a verdict and a reason_code, and pairing the right verdict with the wrong reason still fails. The worker also accepts a JDBC URL, so nothing can be pinned to the shipped store.

Rewrite Worker.run so the decision order follows the policy instead of the registry. Load artifact_evidence for the pending row first and deny revoked signers straight away, before any HTTP call. Then GET /artifacts/{id} purely for the detached_signature, and POST /verify with the digest from H2, never the registry_digest. Map each GET and POST status to its verdict and reason_code from the policy table, MERGE one row per artifact into attestation_reports, and commit once at the end. The key insight is that the registry digest is a decoy: the only thing the lookup contributes is the signature.

Pytest drives the running API and the live H2 store rather than fixtures. Every shipped report is compared against expectations recomputed at test time from real GET and POST calls, so expected verdicts follow whatever the API actually returns. Two tests clone the store, seed a fresh pending row, and rerun the agent-built worker against that JDBC URL, one for an unknown artifact that must be denied and one for a live artifact that must come back trusted, which proves the reports are produced by the program and not written by hand. A separate probe posts art-beta's registry_digest to /verify and shows it comes back 409, so a worker that skipped the H2 digest cannot pass by luck.
