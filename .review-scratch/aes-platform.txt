DIFFICULTY=This task is hard because the rules you need are buried in a long forensic report, not in a summary table. The agent has to extend a Java pipeline that reads those appendices, walks SQLite audit history in recorded_at order with rotation-replacement precedence, pulls ciphertext out of a GIF, and runs AES-256-GCM with frame_id as AAD. Get the key version or nonce wrong on one frame and authentication fails even if the rest of the code is fine.
SOLUTION=I extended the provided three-stage Java CLI under com.mariner.forensic.Main. The rules stage regex-parses Appendix C and D for precedence lists and the frm-003 nonce override. correlate applies that logic to SQLite audit events - rotation replacement for frm-002, override or derived nonce per frame. decrypt extracts ciphertext from the GIF and verifies AES-256-GCM using keys from the key_material table. The oracle compiles to /app/pipeline/classes and runs each stage in order. The main gotcha is frm-002 needs replacement key version 3 after rotation, and frm-003 needs the explicit nonce from the report, not the derived default.
VERIFICATION=Each milestone runs the compiled Main subcommand for that stage and checks JSON against the published schema plus hidden expected files under tests/. M1 checks precedence order, frm-003 override, and rules ground truth - derived_nonce_rule must match the verbatim Appendix C labeled line (trailing period optional). The pipeline must compile Main under /app/classes or /app/pipeline/classes. M2 checks JDBC correlation in recorded_at order, including frm-002 rotation_replacement and frm-003 nonce override. M3 checks schema-valid findings, all three frames with auth_ok and reason_code authenticated, and match to hidden expected_findings.json.
