=== Difficulty ===
The two sources disagree on purpose and each one is canonical for different fields, so the agent cannot shortcut through either of them. The registry API inflates beta's metrics so trusting the API promotes the wrong model, calibration only exists in the H2 store, and the lineage table is version-keyed history — a join on model id alone matches delta's stale 1.0.2 hash and flips the winner to delta. One candidate has no experiment rows at all and must fail closed, while a retired model with the best metrics in H2 is not a registry candidate and must be ignored; two candidates pass every gate so the tie-break and lost_tiebreak bookkeeping actually matter. On top of the reasoning, the jar has to keep working when it is pointed at a different experiment database through its CLI contract, so a hardcoded manifest or fixture-tuned implementation does not survive.

=== Solution ===
Complete the provided Maven skeleton at /app/reconcile-model-release: pull the candidate list from the Spring Boot API with the JDK HTTP client, read validation_metrics, feature_hash_lineage, and calibration_status over JDBC, and evaluate the three policy gates against the H2 canon with the lineage row selected by model id and registry-reported version. Candidates with incomplete evidence are rejected as missing_canonical_evidence without gate evaluation; qualifiers are ranked by canonical AUC with a lexicographic id tie-break and the losers get lost_tiebreak. Every API-vs-DB disagreement on an H2-canonical field is recorded with both values and canonical_source h2. The manifest is emitted deterministically (sorted, no timestamps) to build/release-decision.json, self-validated against the bundled JSON Schema, and the main method honors optional JDBC-URL and output-path arguments so the same jar reconciles any compatible store.

=== Verification ===
Pytest recomputes the expected decision live from the H2 store and the running API — no winner id is hardcoded in the tests — and checks the manifest for schema conformance, the exact policy reason codes per rejected candidate, and the complete conflict enumeration with matching values. It then reruns the agent-built jar twice against the shipped store (explicit args and defaults) to prove the manifest is reproducible by the program, and twice more against verifier-built variant H2 databases (a recalibration and a corrected lineage row) where the correct winner changes, so a hand-written manifest or a jar that replays a precomputed answer fails. The reward is binary through the canonical test.sh block, identical for oracle and agent runs.
