=== Difficulty ===
Agents must reconcile the registry API against H2 while reading a promotion policy whose body text is superseded by amendments. The operative rules require H2 validation_metrics for Gate 1 and tie-break, version-scoped feature-hash lineage, and fail-closed handling for missing experiment rows. The registry deliberately overstates beta's metrics and delta's hash disagrees with the version-keyed lineage row, so body-only implementations promote the wrong model or accept the wrong lineage match.

=== Solution ===
Complete the Maven skeleton at /app/reconcile-model-release: pull candidates from the Spring Boot API, read validation_metrics, feature_hash_lineage, and calibration_status over JDBC, and evaluate the amended three-gate policy with version-scoped lineage and H2-canonical metrics. Record API-vs-H2 disagreements under conflicts, rank qualifiers by H2 AUC with a lexicographic id tie-break, validate against the bundled JSON Schema, and write /app/build/release-decision.json. The main method honors optional JDBC-URL and output-path arguments.

=== Verification ===
Pytest recomputes the expected decision live from the H2 store and running API, checks schema conformance, exact reason codes, conflicts, and the amended-policy trap that beta must fail on H2 metrics while alpha wins. It reruns the agent-built jar against the shipped store and verifier-built variant databases where recalibration or corrected lineage changes the winner, so a hand-written manifest or registry-metric shortcut fails.
