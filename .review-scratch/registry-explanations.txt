=== Difficulty ===
Agents must finish a Java reconcile CLI that joins live HTTP candidates with JDBC evidence while composing four interacting governance systems: operative validation runs with direct and transitive supersession voiding, calibration replay with same-timestamp event_id ordering, waiver event replay with reciprocal replacement pairs, and temporal waiver anchoring that requires the grant to precede the operative run captured_at. The shipped data includes registry metric overstatement, a voided high-water validation run that promotes alpha if void logic is shallow, stale calibration snapshots, post-run waiver grants that look active but cannot suppress, and omega calibration events that share a timestamp — so reading only the policy body or snapshot tables promotes the wrong model.

=== Solution ===
Implement `/app/reconcile-model-release` in Java: fetch registry candidates, derive operative validation metrics after transitive void propagation, replay calibration_events through release_context.decision_at with event_id tie-breaks, join version-scoped lineage, and replay waiver_events with reciprocal replacement validation. Suppress only matching raw gate failures when the grant precedes the operative run, report only grants that were actually used, record H2-canonical conflicts, rank qualifiers by H2 AUC, validate the schema, and write the requested output for either the default or an alternate JDBC store.

=== Verification ===
Pytest recomputes the complete decision from the live API and H2 store, checking schema conformance, transitive operative-run selection, same-timestamp calibration replay, waiver lifecycle and temporal anchoring, exact reason codes, conflicts, and applied_waivers. It reruns the Java CLI against the shipped store and three verifier-built variant databases so hardcoded manifests, snapshot-table shortcuts, shallow void lists, and post-run waiver allow-lists fail.
