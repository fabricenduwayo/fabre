The normalisation rules are not written down anywhere, because there is no specification. They exist only as the behaviour of a compiled jar, and the one corpus the agent can read does not exercise them. The sample has no duplicate groups at all, so the rule that decides which row survives a collapse has zero representation in the evidence available at solve time. What the sample does show is a single mojibake row, which signals that the encoding needs attention without revealing that the decision is taken per file rather than per line. An agent that fixes the encoding until the sample matches gets a clean green light with no signal left, believes it is finished, and ships something that diverges on most rows of every other export. I measured this: an ingester with correct per-file decoding and first-seen collapse matches the sample exactly and then differs on 34 of 49, 34 of 46, and 32 of 40 rows on three held-out corpora. The failure is not a missed clause, it is stopping when the evidence says stop.

Work out what the old ingester does by running it on inputs you build yourself, then reimplement it. Three decisions carry the task. Decode is UTF-8 when the whole file is valid UTF-8 and windows-1252 otherwise, decided once over the entire byte content, so one bad byte near the end changes how the valid multi-byte sequences above it are read. The collapse key is computed after NFKC normalisation, case folding, and internal whitespace collapse, so an encoding misjudgement silently changes which rows are considered the same. Within a collapsed group the surviving row is the one with the greatest seq, not the first seen. Rows that cannot be used are dropped: comments, blanks, wrong field counts, unparseable numbers. Write the survivors into canonical_ledger ordered by seq then account, and emit the per-file and aggregate counts as JSON.

Pytest runs the agent's own program through the entry point the instruction names, and computes the expected answer by running the legacy jar over the same corpus at verify time, so nothing in the verifier encodes the rules. Most tests synthesise a corpus the agent has never seen, each isolating one axis: duplicate groups where first-seen and the legacy choice disagree, rows differing only by case and spacing, a file carrying an undecodable byte after valid multi-byte sequences, and one corpus where all three interact. Two further tests check the ingester fills whichever store it is given and that a repeated run leaves the ledger unchanged. Before the suite starts, test.sh moves the legacy jar out of /opt/legacy, which the instruction already warned would happen, so parity has to come from the agent's own code rather than a shell-out.
